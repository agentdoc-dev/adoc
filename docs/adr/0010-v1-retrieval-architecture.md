# Adopt artifact-only retrieval, sidecar embedding artifact, and hybrid ranking for V1

V1 turns the V0 compiler output into a usable local retrieval surface for humans and for any agent that already knows Object IDs from the agent artifact. This ADR fixes three coupled architecture choices made during V1 brainstorming so that the V1-DESIGN contract has a single record to point at: V1 retrieval is artifact-only, embeddings ship in V1 as a separate sidecar artifact, and ranking is a fixed-formula hybrid with no tunable score weights.

The first decision is artifact-only retrieval. `adoc explain` and `adoc search` read `dist/docs.agent.json` (and, when present, `dist/docs.search.json`) and never re-run `compile_workspace()`. This preserves the V0 thesis that source is canonical and the agent artifact is a derived read model: retrieval commands read the latest build, and a missing or stale build is the user's responsibility, surfaced via fix-oriented diagnostics that point at `adoc build`. Source-aware retrieval, watch mode, and incremental partial recompilation are deferred to V1.5+ where they fit alongside the project initializer and minimal config story. The V0 boundary between `adoc check`/`adoc build` (write side) and `adoc explain`/`adoc search` (read side) is intentional and load-bearing.

The second decision is to ship embeddings in V1 instead of deferring them past exact search. The original ROADMAP staged embeddings well after V1 because lexical search was deemed sufficient until proven otherwise. We are reversing that staging because (a) the V0 thesis is "agents retrieve typed knowledge objects", and the only retrieval surface that delivers on that thesis with paraphrase recall is semantic; (b) the embedding subsystem is the single largest dependency-and-architecture decision in V1, and shipping it inside V1 forces the contract to be settled while the rest of the retrieval pipeline is still small enough to reshape; (c) waiting until V2 leaves a real product gap that any external consumer would have to fill with an ad-hoc index. The trade is one extra slice's worth of work and one new internal port.

The third decision is the hybrid ranking formula. V1 ranks via Reciprocal Rank Fusion (`k = 60`) over a BM25 lexical index (rebuilt at session load over `body`, `id`, `kind`, `owner`) and a brute-force cosine vector index (loaded from the search artifact). Exact Object ID and ID-prefix matches are pinned above the fused list. RRF is chosen specifically because it is parameter-free in practice: one constant `k` with a well-known default, no per-feature weights, no per-corpus tuning. The PRD's full multi-factor scoring set (lifecycle, freshness, evidence quality, owner authority, contradiction state, usage history) is not in V1 - those are filters in V1, never score modifiers, and they remain filter-only until we have real retrieval evaluation data to set weights against. This is the same discipline as ADR-0007: a small, justifiable set of behaviors first, expansion only when an observed failure mode demands it.

The embedding compute boundary is a new internal port, `EmbeddingProvider`, governed by ADR-0006. The default adapter wraps `fastembed-rs` with `bge-small-en-v1.5` and runs locally; first run downloads weights via the crate's HF fetcher, subsequent runs are offline. A deterministic `InMemoryProvider` adapter is used by every hermetic test, mirroring the role `InMemorySourceProvider` plays today. A hosted adapter (OpenAI, Voyage, Cohere) is explicitly possible later as a third file in `infrastructure/embedding/` and one CLI flag, with no port churn. The port stays `pub(crate)` until an external consumer (LSP, MCP server, alternate CLI) needs it.

The embedding artifact shape is a sidecar JSON, `dist/docs.search.json`, with a model header (`{ id, provider, dim }`), a hash of the agent artifact for drift detection, and one `{ id, content_hash, vector }` entry per Knowledge Object. Per-Object-ID `content_hash` is computed over a canonical embedding-input string (`{kind}: {body_plain_text}\n[id: ...] [status: ...] [owner: ...]`) and gates an in-build cache: when content hashes agree across builds, prior vectors are reused. The composition formula is part of the contract; changing it requires bumping `adoc.search.v0` to `adoc.search.v1` and invalidating prior search artifacts. Sidecar JSON is the simplest viable shape - git-diffable, inspectable, no new deps beyond `serde`, and well within performance budget at pilot scale (10k objects × 384 dim ≈ 15 MB f32 raw, ≈ 50 MB JSON, sub-50ms cold load and brute-force query). Migration to a binary sidecar or to SQLite + sqlite-vec is a future concern, not a V1 one. Vectors are not embedded inside `docs.agent.json`: that contract is reserved for the V1 read model and must not bloat with retrieval-only payload.

This ADR preserves and refines its predecessors. ADR-0001 (`adoc` CLI command), ADR-0002 (Rust for V0), ADR-0003 (two-crate workspace), ADR-0004 (hand-written parser), and ADR-0005 (single public `compile_workspace` entry) are unchanged in V1; the new public functions `load_retrieval_session`, `explain_object`, and `search` are read-side additions that do not break the V0 entry point. ADR-0006 is preserved with one new port (`EmbeddingProvider`) added under the same `pub(crate)` rule. ADR-0007 is preserved: every new diagnostic code (`embed.*`, `io.artifact_*`, `search.*`, `retrieval.object_not_found`, `schema.unsupported_version`, `id.duplicate_in_artifact`, `build.embeddings_skipped`) follows the typed-emission rule and lands as a stable semantic identifier rather than a numeric alias. ADR-0008 is preserved: hermetic tests use `InMemoryProvider`; the fastembed path runs under a gated feature flag so default `cargo test` does not pull weights. ADR-0009 is preserved: pure retrieval domain types live in `domain/retrieval/`, embedding adapters live in `infrastructure/embedding/`, the application layer composes ports in `application/retrieval.rs`, and `lib.rs` remains the composition root.

## Addendum: scope of the V1 retrieval surface

The V1 retrieval surface is two CLI commands (`adoc explain`, `adoc search`) and three additions to the public core (`load_retrieval_session`, `explain_object`, `search`). It is not a server, not a library API for parsing or rendering, and not a programmatic patching API. The `--format json` envelope on both commands (`adoc.retrieval.v0`) is the wire format an external server would later wrap; choosing it now and pinning its shape inside V1 keeps a future MCP wrapper a thin shim rather than a contract negotiation. The search-artifact contract (`adoc.search.v0`) and the retrieval envelope (`adoc.retrieval.v0`) are versioned independently from the agent artifact (`adoc.agent.v0`) so that V0 consumers are not forced to migrate when retrieval evolves.

## Addendum: where filters live in V1 and what they cost later

V1 filters are `--kind`, `--status`, `--owner`, `--source-path`. They apply pre-rank in `--lexical` and `--semantic` modes (the candidate pool is small enough that filtering first is cheap and improves recall) and post-rank in `--hybrid` (so RRF compares a comparable candidate pool from both indexes). The PRD's wider filter set - scope, evidence type, changed-since, agent visibility, permissions - is intentionally absent in V1 because each one requires an upstream contract that does not exist yet (scope schema, evidence object kinds, source-path impact analysis from V3, permission model from V7). Adding any of those filters in V1 would either bake in a guess at the upstream contract or ship a filter that silently no-ops on missing fields. Both are worse than waiting.

## Addendum: model-swap and re-embed semantics

The search artifact's model header is the contract for invalidation. When the active provider's `(provider, id, dim)` differs from the artifact's, retrieval refuses to use the search artifact and emits `search.model_mismatch`; rebuilding with the new provider regenerates every entry. Within the same model, the per-object `content_hash` gates re-embed; bodies and metadata that did not change are not re-encoded. The `agent_artifact_hash` field captures whole-artifact drift: when the agent artifact has been rebuilt without rebuilding the search artifact, retrieval still works but emits `search.hash_drift` (warn) so users know they may be querying stale embeddings. This is intentionally lenient - hard-failing a query because the search artifact is two minutes older than the agent artifact would surprise users mid-edit. Rebuilding both artifacts together is the normal path; the warning is the safety net.
