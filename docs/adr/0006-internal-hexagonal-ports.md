# Use internal hexagonal ports for sources, rendering, and artifacts

AgentDoc V0 keeps its public surface to `compile_workspace()` (per ADR-0005) but factors the orchestrator's collaborators into `pub(crate)` ports inside `adoc-core`: `SourceProvider` for input discovery (with `FsSourceProvider` and a test-only `InMemorySourceProvider`), `Renderer` for human-facing output (with `HtmlRenderer`), and `ArtifactWriter` for typed agent artifacts (with `AgentJsonArtifact`). This makes the orchestrator's error-gating logic unit-testable without touching the filesystem, removes the open-closed violation that previously required editing `compile.rs` to add a new format, and leaves the door open for future LSP, web-preview, or semantic-diff consumers to reuse the adapters without reopening this ADR. Adapters are dispatched statically (no `dyn`) so the abstraction is free at runtime.

`ArtifactWriter` exposes its return shape as an associated `type Output` (not a hard-coded `AgentJsonDocument`) so a future `RdfArtifact` or `MarkdownIndexArtifact` slots in by declaring its own `Output` — proving the open-closed claim. The trait method is named `build` rather than `write`: it constructs the in-memory document; the CLI owns the file-write boundary. A test stub (`CountingArtifact` with `type Output = String`) pins the format-agnostic contract so a future regression that re-locks the return type is caught by `cargo test` rather than by a downstream review.

## Addendum: the orchestrator is itself a structural seam

`compile_with_provider` is split into named pipeline stages: `load_pages` (provider → parsed (source, page) pairs + io/parse diagnostics), `validate_pages` (per-page rules over the pairs), `assemble_workspace` (drops sources, moves pages into the `WorkspaceAst` aggregate), `validate_workspace` (workspace-level rules), and `build_artifacts` (gates the renderer + agent-JSON pair on diagnostic severity). Each stage is independently testable (see `#[cfg(test)]` `load_pages_*`, `validate_pages_*`, `assemble_workspace_*`, `build_artifacts_*` in compile.rs); the orchestrator becomes a five-line sequence of named domain operations rather than one walls-of-text loop. Pages move through the pipeline without cloning. Adding a new stage (e.g. a future link-graph build) is a new function plus a new line in `compile_with_provider`, not another `for` loop body.
