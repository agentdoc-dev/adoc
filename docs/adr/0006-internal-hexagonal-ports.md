# Use internal hexagonal ports for sources, artifacts, and embeddings

AgentDoc keeps its public compiler surface small (per ADR-0005) but factors application collaborators into `pub(crate)` ports inside `adoc-core`: `SourceProvider` for input discovery, `ArtifactWriter` and `ArtifactReader` for structured build/read artifacts, and `EmbeddingProvider` for local vector generation. `HtmlRenderer` remains an infrastructure adapter selected by the compile application service; it renders from `WorkspaceAst` or private render projections, not from serialized graph JSON. This keeps orchestration testable without touching the filesystem and leaves the door open for future LSP, web-preview, or semantic-diff consumers to reuse adapters without reopening this ADR. Adapters are dispatched statically where possible; the embedding provider uses a trait object at the factory boundary because provider selection is runtime configuration.

`ArtifactWriter` exposes its return shape as an associated `type Output` (not a hard-coded public graph document) so a future `MarkdownIndexArtifact` or binary sidecar slots in by declaring its own `Output` - proving the open-closed claim. The trait method is named `build` rather than `write`: it constructs the adapter-local in-memory output; the application layer serializes supported artifacts into `BuildArtifacts`, and the CLI owns the file-write boundary. A test stub (`CountingArtifact` with `type Output = String`) pins the format-agnostic contract so a future regression that re-locks the return type is caught by `cargo test` rather than by a downstream review.

## Addendum: the orchestrator is itself a structural seam

`compile_with_provider` is split into named pipeline stages: `load_pages` (provider -> parsed (source, page) pairs + io/parse diagnostics), `validate_source_pages` (pre-resolution source rules over parser-owned spans), `resolve_knowledge_objects` (pending typed blocks -> typed aggregates or diagnostics), `resolve_object_references` (pending object references and relation targets -> resolved references or diagnostics), `validate_resolved_pages` (post-resolution page rules), `assemble_workspace` (drops sources, moves pages into the `WorkspaceAst` aggregate), `validate_workspace` (workspace-level rules), and `build_artifacts` (gates renderer and artifact emission on diagnostic severity). Each stage is independently testable (see `#[cfg(test)]` `load_pages_*`, `validate_source_pages_*`, `resolve_knowledge_objects_*`, `resolve_object_references_*`, `validate_resolved_pages_*`, `assemble_workspace_*`, `build_artifacts_*`); the orchestrator becomes a short sequence of named domain operations rather than one walls-of-text loop. Pages move through the pipeline without cloning. Adding a new stage is a new function plus a new line in `compile_with_provider`, not another `for` loop body.

## Addendum: domain projections are not output ports

`domain/knowledge_object/projection.rs` now owns the Knowledge Object metadata projection consumed by `HtmlRenderer` and `GraphJsonArtifact`. This does not weaken the static, format-locked port decision above: the projection is an internal domain query shape over aggregates, not a dynamic renderer/artifact port and not a public output format. Format adapters still own markup, JSON keys, escaping, ordering, and file-write boundaries; the projection only prevents adapters from rematching aggregate internals to rediscover metadata.

## Addendum: graph and search DTOs are adapter internals

The Graph Artifact is presentation-free data. `GraphJsonArtifact` does not import HTML rendering and does not serialize `html` fields on graph nodes; HTML output is produced separately by `HtmlRenderer`. The graph/search artifact DTOs remain `pub(crate)` so Rust callers cannot couple themselves to construction structs that may change while `adoc.graph.v1` and `adoc.search.v0` are still settling. Public callers receive ready-to-write artifact strings and retrieval envelopes instead.
